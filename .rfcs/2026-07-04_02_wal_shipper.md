# RFC: in-BEAM WAL shipper (litestream replacement)

**Status: design spike — prototype and go/no-go still open.**

## Why

The phase 3 density benchmark put per-database litestream overhead at
~400KB (sidecar registration, per-db shadow WAL, S3 layout), failing
the 100k dbs/node target for universal enablement — which is why
continuous replication is a per-database premium flag today. An
in-BEAM shipper targets ~KB/db bookkeeping, no sidecar, and one code
path for all tiers. This RFC fixes the design questions; a throwaway
prototype must produce numbers before any commitment.

## Design sketch

**Capture.** Each hot `Database.Server` already owns the sole writer
connection. Two capture options:

1. *WAL tailing* (litestream's approach): read `<db>-wal` frames after
   each transaction; requires disabling auto-checkpoint and running
   controlled checkpoints (`PRAGMA wal_checkpoint(TRUNCATE)`) only
   after shipped offsets are durable.
2. *Commit-hook capture*: exqlite exposes `set_update_hook`, but that
   yields row-level events, not WAL frames — rebuilding pages from row
   events is a non-starter. WAL tailing wins.

Tail in the Server process itself (it serializes writes, so frame
boundaries are quiescent points): after a write-classified... after
*any* successful statement, stat the WAL; if it grew, read new frames
and append to the per-database ship buffer. No extra process per
database — an on-node `WalShipper` GenServer pool batches uploads
across databases (the density win: idle databases cost zero).

**Object format.** `wal/<tenant>/<db>/<generation>/<start_offset>.seg`
— length-prefixed frame runs with a header {generation, page_size,
salt1, salt2, start_frame, crc}. `generation` reuses the
idle-snapshot generation counter: a full snapshot (idle-stop ship or
first activation) starts a generation; segments apply on top of the
snapshot with the matching generation. Restore = fetch
`idle-snapshots/.../latest.db` + replay segments of that generation in
offset order (VFS-free: write frames into a reconstructed `-wal` and
let SQLite recover, exactly like litestream restore).

**Checkpoint coordination.** The single-writer property makes this
tractable: the Server performs checkpoints itself, only when the
shipper confirms all frames below the checkpoint mark are in S3. WAL
growth between confirmations is bounded by a max-WAL-bytes knob that
degrades to "block writes until shipped" (backpressure, never data
loss).

**Torn writes / crash windows.** Segments are immutable and
content-addressed by offset; a crashed upload re-uploads the same
range idempotently. A crash between commit and ship loses at most the
unshipped tail — same RPO shape as litestream's sync interval,
tunable per tier via ship-interval limits.

**Failover.** Same as today: restore = snapshot + segments. The
generation marker scheme already guards stale caches; segment replay
extends `snapshot_generation` with a within-generation offset.

## What the prototype must measure (exit criteria)

Reuse the phase 3 density bench harness:

1. Per-hot-database memory/CPU overhead of tailing at 100k dbs/node
   (target: <10KB per idle-registered db, <5% CPU at the phase 3 write
   mix).
2. Ship latency p99 at 1k concurrent writers (target: ≤ litestream's
   measured 1s sync interval behavior).
3. `kill -9` during sustained writes × 100 iterations: restore
   correctness (checksum equality against a synchronous copy) — zero
   tolerance.
4. S3 request amplification vs litestream at the same workload
   (batched multi-db uploads should win; verify).

## Non-goals

Point-in-time restore beyond segment granularity, cross-region fanout,
and replacing manual backups — those stay as-is.

## Go / no-go

Open. No-go conditions: prototype misses the overhead target by >2x,
or any restore-correctness failure without a clear fix. Until go, the
per-database litestream premium flag remains the crash-window answer.
